## How It Works

### The Core Loop

```
for each story in prd.json (by priority):
    for each attempt (up to MAX_RETRIES):
        for each step in the story's pipeline:
            run step (fresh agent instance, reads prompt file)
            if step says REVISE → break, retry from first step
        if all steps pass → mark story as passed, next story
    if max retries hit → stop with error
```

### Per-Iteration Flow

Each story has a pipeline (e.g., `["work", "review"]`). Each step is
a fresh agent instance (Claude or Codex) with no memory of prior steps — state is
communicated through files.

```
┌─────────────── Iteration N ───────────────┐
│                                           │
│  STEP 1: work                             │
│  → Fresh agent reads work.md prompt       │
│  → Reads current-story.txt + prd.json     │
│  → Reads review-feedback.txt (if retry)   │
│  → Implements code, runs tests, commits   │
│  → Writes work-summary.txt               │
│                                           │
│  STEP 2: review                           │
│  → Fresh agent reads review.md prompt     │
│  → Reads what worker did                  │
│  → Runs tests independently               │
│  → Checks for cheating/shortcuts          │
│  → Writes SHIP or REVISE to              │
│    review-result.txt                      │
│                                           │
│  STEP 3: visual-verify (if in pipeline)   │
│  → Fresh agent with browser MCP           │
│  → Takes screenshots, verifies UI         │
│  → Writes SHIP or REVISE                  │
│                                           │
│  Any REVISE → retry from step 1           │
│  All SHIP → story passes, move on         │
└───────────────────────────────────────────┘
```

### State Management

State lives in FILES, not in the LLM's memory. Each agent instance
(Claude or Codex) starts fresh and reads from:

| File                           | Purpose                           |
| ------------------------------ | --------------------------------- |
| `.gyro/state/current-story.txt` | Which story ID to work on       |
| `.gyro/state/work-summary.txt`  | What the worker did             |
| `.gyro/state/review-result.txt` | SHIP or REVISE                  |
| `.gyro/state/review-feedback.txt` | Reviewer's complaints         |
| `.gyro/prd.json`               | All stories + pass/fail status  |
| `.gyro/progress.txt`           | Append-only completion log      |
| `.gyro/learnings.md`           | Operational learnings (mistakes & gotchas) |
| `.gyro/specs/*.md`             | Requirements source of truth    |
| `AGENTS.md`                     | Project rules + quality gates   |
| `PLAN.md`                       | Planning mode output            |
| Git history                     | Record of all completed work    |

### Why Fresh Context Matters

Traditional agent loops accumulate context — failed attempts, dead
ends, noise — which degrades quality. The Gyro Loop resets context
every step but preserves state externally. Each new instance starts
clean but has full visibility into prior work through the filesystem
and git.


## Quick Start

```bash
# 1. Make sure you have at least one AI CLI
claude --version    # Claude Code CLI
codex --version     # OpenAI Codex CLI (alternative)
jq --version        # JSON processor

# 2. Review the execution plan
./gyro.sh --dry-run

# 3. Run it
./gyro.sh

# 4. If a story fails, fix the issue and resume
./gyro.sh --from story-05
```


## File Structure

```
project/
├── gyro.sh                    # Main orchestrator script
├── AGENTS.md                   # Project conventions + quality gates
├── PLAN.md                     # Generated by --plan mode
└── .gyro/
    ├── prd.json                # Product requirements (stories + models)
    ├── progress.txt            # Append-only log of completed work
    ├── gyro.log               # Orchestrator debug log
    ├── learnings.md            # Operational learnings (append-only)
    ├── prompts/                # One .md file per pipeline step
    │   ├── work.md             # Worker: implements code
    │   ├── review.md           # Reviewer: catches cheating/bugs
    │   ├── visual-verify.md    # Visual QA: browser screenshots
    │   ├── simplify.md         # Simplification pass
    │   ├── plan.md             # Planning mode: gap analysis
    │   └── push.md             # Push to remote (standalone checkpoint)
    ├── specs/                  # Requirements source of truth
    │   ├── README.md           # Specs convention guide
    │   └── *.md                # One file per topic of concern
    └── state/                  # Per-iteration state (managed by gyro.sh)
        ├── current-story.txt
        ├── work-summary.txt
        ├── review-result.txt
        ├── review-feedback.txt
        ├── checkpoint-scope.txt # Git tag for scoped checkpoints
        └── *-output.log        # Raw output from each step
```


## PRD Format

The `prd.json` file defines your stories, pipelines, and checkpoints.

### Minimal Example

```json
{
  "project": "my-app",
  "pipelines": {
    "default": ["work", "review"]
  },
  "models": {
    "work": "codex:gpt-5.3-codex",
    "review": "claude:opus"
  },
  "stories": [
    {
      "id": "story-01",
      "title": "Add login endpoint",
      "pipeline": "default",
      "acceptance_criteria": [
        "POST /login with valid credentials returns 200 + JWT token",
        "POST /login with bad password returns 401"
      ],
      "passes": false,
      "priority": 1
    }
  ]
}
```

### Full Example (with checkpoints + model selection)

```json
{
  "project": "task-crud-app",
  "models": {
    "work": "codex:gpt-5.3-codex",
    "review": "claude:opus",
    "simplify": "claude:sonnet",
    "visual-verify": "claude:haiku",
    "plan": "claude:opus"
  },
  "pipelines": {
    "setup":        ["work", "review"],
    "backend-tdd":  ["work", "review"],
    "frontend-e2e": ["work", "review", "visual-verify"],
    "simplify":     ["simplify", "review"]
  },
  "checkpoints": {
    "simplify": {
      "after": ["story-06", "story-11"],
      "on_complete": true
    }
  },
  "stories": [
    {
      "id": "story-01",
      "title": "Initialize project",
      "pipeline": "setup",
      "acceptance_criteria": ["..."],
      "passes": false,
      "priority": 1
    },
    {
      "id": "story-02",
      "title": "POST /tasks (TDD)",
      "pipeline": "backend-tdd",
      "acceptance_criteria": [
        "Write test FIRST: POST /tasks with {title:'Buy milk'} returns 201",
        "Write test: empty title returns 400",
        "Implement handler to make tests pass",
        "go test ./... passes"
      ],
      "passes": false,
      "priority": 2
    }
  ]
}
```

### Story Fields

| Field                | Required | Description                                   |
| -------------------- | -------- | --------------------------------------------- |
| `id`                 | yes      | Unique identifier (e.g., `story-01`)          |
| `title`              | yes      | Short description of the task                 |
| `pipeline`           | yes      | Which pipeline to use (must exist in pipelines)|
| `acceptance_criteria` | yes     | List of verifiable criteria                   |
| `passes`             | yes      | `false` initially, `true` when shipped        |
| `priority`           | yes      | Lower number = higher priority, done first    |

### Writing Good Acceptance Criteria

Criteria must be specific and verifiable. Vague criteria get gamed.

```
BAD:   "Validates input"
GOOD:  "POST /tasks with empty title returns 400 with {error: 'title is required'}"

BAD:   "Returns tasks"
GOOD:  "GET /tasks after creating 3 tasks returns 200 with array of length 3,
        each containing id (int), title (string), and createdAt (ISO 8601)"

BAD:   "Looks good"
GOOD:  "Form has text input with placeholder 'Task title' and button labeled 'Add Task'"
```


## Pipelines

Pipelines define the sequence of steps for a story type. Each step
maps to a prompt file in `.gyro/prompts/{step-name}.md`.

### Built-in Pipelines

```json
{
  "setup":        ["work", "review"],
  "backend-tdd":  ["work", "review"],
  "frontend-e2e": ["work", "review", "visual-verify"],
  "simplify":     ["simplify", "review"]
}
```

### How Steps Are Executed

1. gyro.sh reads the story's pipeline
2. For each step, it pipes the matching prompt file to agent (Claude or Codex)
3. After each step, it checks for `.gyro/state/review-result.txt`
4. If `REVISE` → abort remaining steps, retry from step 1
5. If `SHIP` → clear the result file, continue to next step
6. If all steps complete without `REVISE` → story passes

### Creating Custom Pipelines

Add a new pipeline by:
1. Adding the pipeline name + steps to `prd.json`
2. Creating a prompt file for each new step

Example — add a security scanning step:

```json
"pipelines": {
  "secure-api": ["work", "review", "security-scan"]
}
```

Then create `.gyro/prompts/security-scan.md` (see Extending section).


## Checkpoints

Checkpoints run between stories at specific points. They're for
cross-cutting concerns that span multiple stories (simplification,
refactoring, performance audits).

### Configuration

```json
"checkpoints": {
  "simplify": {
    "after": ["story-06", "story-11"],
    "on_complete": true
  },
  "push": {
    "after": ["story-11"],
    "on_complete": true,
    "standalone": true
  }
}
```

| Field         | Description                                          |
| ------------- | ---------------------------------------------------- |
| `after`       | Story IDs array, or `"each"` to run after every story |
| `on_complete` | Run after ALL stories pass (final pass)              |
| `standalone`  | Run once with no review loop (e.g., push)            |

### Checkpoint Types

**Reviewed checkpoints** (default): Run the step, then review in a
retry loop — same as story pipelines. Used for simplify, security, etc.

**Standalone checkpoints** (`"standalone": true`): Run the step once,
no review. Used for mechanical tasks like push that don't need AI review.

### Scoped Checkpoints

Checkpoints are scoped using git tags. Each checkpoint type tracks
its own scope independently:

- After a checkpoint succeeds, gyro.sh creates a git tag:
  `gyro-cp-{name}-{N}` (e.g., `gyro-cp-simplify-1`)
- Before the next run, gyro.sh writes the last tag to
  `.gyro/state/checkpoint-scope.txt`
- The checkpoint prompt reads the scope file and only reviews files
  changed since the last tag

This means the second simplify run only reviews code from stories 7-11,
not the entire codebase — stories 1-6 were already simplified.

```
story-01 → commit
...
story-06 → commit
    ↓
CHECKPOINT: simplify  (scope: full codebase)  → tag gyro-cp-simplify-1
    ↓
story-07 → commit
...
story-11 → commit
    ↓
CHECKPOINT: simplify  (scope: since tag -1)   → tag gyro-cp-simplify-2
CHECKPOINT: push      (scope: since start)    → tag gyro-cp-push-1
    ↓
FINAL CHECKPOINT: simplify  (scope: since tag -2)   → tag gyro-cp-simplify-3
FINAL CHECKPOINT: push      (scope: since tag -1)   → tag gyro-cp-push-2
    ↓
Done!
```

Each checkpoint type tracks its own tags independently — simplify
scope is separate from push scope, security scope, etc.

### Checkpoint Behavior

- By default, checkpoints run in a review loop (step + review, up to MAX_RETRIES)
- Standalone checkpoints run once with no review
- If a checkpoint step writes `NO_CHANGES` to work-summary.txt,
  it's considered clean (still gets tagged)
- If a checkpoint fails after all retries, it warns but continues
  (doesn't block the next story)
- All checkpoints create a git tag on success for scope tracking


## Work Branches

When enabled, each story gets its own git branch. This gives you
per-story pull requests for code review, cleaner git history, and
easier rollback.

### Enabling Work Branches

Add `"work_branches": true` to your `prd.json`:

```json
{
  "project": "my-app",
  "work_branches": true,
  "pipelines": { "..." },
  "stories": [ "..." ]
}
```

### How It Works

Branches chain sequentially — each story branches from the previous
story's tip, so it inherits all prior work:

```
main
 └─ gyro/story-01  (branched from main)
      └─ gyro/story-02  (branched from story-01)
           └─ gyro/story-03  (branched from story-02)
```

The loop doesn't wait for merges. It creates branches and pushes as
it goes. You merge the PRs in order after reviewing.

### How Dependencies Are Handled

Since stories run sequentially (sorted by priority), each new branch
starts from the current HEAD — which includes all prior stories' work.
Story-02 can use code from story-01 because it branched from story-01's
tip.

### Pull Request Strategy

All PRs target `main`. Merge them in order:

1. Merge `gyro/story-01` → `main`
2. Now `gyro/story-02`'s PR diff auto-reduces to just its own changes
   (because story-01's code is already in main)
3. Merge `gyro/story-02` → `main`
4. Repeat

### Per-Story Push with `"after": "each"`

To push and create a PR after every story, use `"after": "each"`:

```json
"checkpoints": {
  "push": {
    "after": "each",
    "standalone": true
  }
}
```

This is equivalent to listing every story ID in the `after` array.

### Typical Configuration

```json
{
  "project": "my-app",
  "work_branches": true,
  "checkpoints": {
    "push": {
      "after": "each",
      "standalone": true
    },
    "simplify": {
      "after": ["story-06"],
      "on_complete": true
    }
  }
}
```

This creates a branch + push + PR for each story, plus simplification
at a natural boundary and at the end.

### Resume with Work Branches

When resuming with `--from story-05`, make sure your HEAD contains
all prior stories' work. If `gyro/story-04` exists, check it out
first:

```bash
git checkout gyro/story-04
./gyro.sh --from story-05
```

### Environment Variables

| Variable             | Default | Description                        |
| -------------------- | ------- | ---------------------------------- |
| `GYRO_BASE_BRANCH`  | `main`  | Target branch for pull requests    |


## Prompts

Prompts are Markdown files in `.gyro/prompts/`. Each one is piped
directly to the agent CLI (Claude or Codex) as the full instruction for that step.

### Prompt Design Principles

1. **Tell the agent what it is**: "You are a developer" vs "You are a reviewer"
2. **Tell it what to read first**: List the state files to check
3. **Give explicit rules**: What to NEVER do, what to ALWAYS do
4. **Define the output contract**: What files to write when done
5. **Make the review decision binary**: SHIP or REVISE, nothing else

### Prompt Language Patterns

Specific phrasing makes a measurable difference in agent behavior.
These patterns are battle-tested:

| Pattern                            | Why it works                                           |
| ---------------------------------- | ------------------------------------------------------ |
| **"Study"** not "read" or "look at" | Triggers deeper analysis instead of surface scanning   |
| **"Don't assume not implemented"** | Prevents duplicate code — forces codebase search first |
| **"If functionality is missing, it's your job to add it"** | Eliminates placeholder/stub excuses |
| **"Capture the why"**             | Tests and comments document reasoning, not just behavior |
| **"Using parallel subagents"**    | Explicit instruction to parallelize expensive reads     |
| **"Only 1 subagent for builds/tests"** | Prevents parallel test conflicts               |

### work.md — What It Does

- Reads the assigned story from prd.json
- Checks for review feedback from prior attempt
- Implements the code following project rules
- Runs quality gates
- Commits the change
- Writes summary to work-summary.txt

### review.md — What It Does

- Reads what the worker claims to have done
- Checks every test for gaming (assert true, toBeDefined, etc.)
- Checks every implementation for stubs/hardcoded values
- Runs quality gates independently
- Verifies each acceptance criterion is met
- Writes SHIP or REVISE + specific feedback

### plan.md — What It Does

- Reads specs/ and prd.json for requirements
- Explores current codebase
- Does gap analysis: what specs say vs what code does
- Outputs/updates `PLAN.md` with prioritized tasks
- Does NOT implement anything

### visual-verify.md — What It Does

- Starts dev servers
- Uses browser MCP to navigate to the app
- Takes screenshots, saves to `.gyro/screenshots/{story-id}/`
- Preserves prior screenshots for comparison across runs
- Verifies UI matches acceptance criteria
- Writes SHIP or REVISE

### push.md — What It Does

- Verifies clean working tree
- Pushes the current branch to remote
- Creates a pull request via `gh pr create` (if `gh` CLI available)
  - PR includes story title, acceptance criteria, and work summary
  - Targets `main` (or `GYRO_BASE_BRANCH`)
- Never force-pushes
- Runs as a standalone checkpoint (no review loop)

### simplify.md — What It Does

- Reads changed files since last simplify tag (or full codebase on first run)
- Looks for duplication, dead code, over-engineering
- Makes simplification changes
- Ensures all tests still pass
- Commits small, focused changes
- Writes NO_CHANGES if nothing needs simplifying

## Model Selection

Each pipeline step can use a different agent and model. Configure
this with the `models` field in `prd.json` using `"agent:model"` format:

```json
{
  "models": {
    "work": "codex:gpt-5.3-codex",
    "review": "claude:sonnet",
    "simplify": "claude:sonnet",
    "visual-verify": "claude:haiku",
    "plan": "claude:opus"
  }
}
```

### Format

```
"agent:model"   — explicit agent and model (e.g., "codex:gpt-5.3-codex", "claude:opus")
"model"         — bare alias, uses default agent (e.g., "opus", "sonnet")
```

This lets you mix agents per step — use Codex for heavy implementation
and Claude for review, or vice versa.

### How It Works

1. When `run_step()` executes, it reads the `models` config for that step
2. If the value contains `:`, it splits into agent + model
3. If no `:`, it uses the default agent (`GYRO_DEFAULT_AGENT` env var, auto-detected)
4. The model alias is expanded to the correct model ID for that agent
5. The appropriate CLI command is built and executed

### Supported Agents

| Agent    | CLI Command                                         |
| -------- | --------------------------------------------------- |
| `claude` | `claude -p --dangerously-skip-permissions -m MODEL` |
| `codex`  | `codex exec -s danger-full-access -m MODEL`         |

### Supported Aliases

Aliases are expanded per agent:

| Alias    | Claude Code                      | Codex CLI    |
| -------- | -------------------------------- | ------------ |
| `opus`   | `claude-opus-4-6`               | `gpt-5.3-codex` |
| `sonnet` | `claude-sonnet-4-6`             | `gpt-5.3-codex` |
| `haiku`  | `claude-haiku-4-5-20251001`     | `gpt-5.3-codex` |

You can also use full model IDs directly (e.g., `"codex:gpt-5.3-codex"`,
`"claude:claude-sonnet-4-6"`).

### Available Codex Models (ChatGPT Login)

Tested on **March 4, 2026** with `codex-cli 0.107.0` and `codex login status = Logged in using ChatGPT`.

| Model ID          | Status in this setup            | Notes |
| ----------------- | ------------------------------- | ----- |
| `gpt-5.3-codex`   | Works by default                | Recommended default in this repo |
| `gpt-5-codex`     | Works with config change        | Needs `model_reasoning_effort = "high"` (or `medium`/`low`) |
| `gpt-5`           | Works with config change        | Needs `model_reasoning_effort = "high"` (or `medium`/`low`/`minimal`) |
| `o3`              | Not supported                   | Fails for ChatGPT-authenticated Codex |
| `o4-mini`         | Not supported                   | Fails for ChatGPT-authenticated Codex |
| `gpt-4.1`         | Not supported                   | Fails for ChatGPT-authenticated Codex |
| `gpt-5-mini`      | Not supported                   | Fails for ChatGPT-authenticated Codex |
| `gpt-5-nano`      | Not supported                   | Fails for ChatGPT-authenticated Codex |

To switch models, set the step value in `.gyro/prd.json`:

```json
{
  "models": {
    "work": "codex:gpt-5.3-codex"
  }
}
```

If you use `gpt-5-codex` or `gpt-5`, set this in `~/.codex/config.toml`:

```toml
model_reasoning_effort = "high"
```

## Specs Directory

The `.gyro/specs/` directory holds requirement specifications — the
source of truth for what the project should do.

### Convention

- **One file per topic of concern** (e.g., `authentication.md`, `filtering.md`)
- **Scope test**: each file's topic should be describable in one
  sentence without using "and"
- **Behavioral outcomes**, not implementation details
- **Markdown format** with testable criteria

### Example

`.gyro/specs/todo-crud.md`:
```markdown
# Todo CRUD

## Overview
Users can create, read, update, and delete todo items.

## Requirements
- Adding a todo with text creates it in the list
- Empty text is rejected (no empty todos)
- Each todo can be toggled between active and completed
- Deleting a todo removes it permanently

## Edge Cases
- Very long todo text (>500 chars) is truncated
- Special characters in text are preserved
```

### How Specs Are Used

All prompts (work, review, simplify, visual-verify, plan) check for
the specs directory at startup. If it exists:

- **plan** reads ALL specs for gap analysis
- **work** reads relevant specs for detailed requirements context
- **review** checks implementation against specs
- **simplify** ensures simplifications don't violate specs
- **visual-verify** reads specs for visual/UX requirements

Specs complement `prd.json` — specs are the richer requirements docs,
`prd.json` stories are the execution units derived from them.

### Workflow: Specs → Plan → Stories

```bash
# 1. Write specs (human)
vim .gyro/specs/authentication.md

# 2. Run planning mode (AI reads specs, analyzes code, writes PLAN.md)
./gyro.sh --plan

# 3. Convert plan to stories (AI or human)
./plan-to-gyro.sh PLAN.md

# 4. Run the loop
./gyro.sh
```


## Learnings

The `.gyro/learnings.md` file captures operational mistakes and
gotchas so future iterations don't repeat them.

### How It Works

Every prompt reads `learnings.md` at startup. When a step hits a
problem, it appends a one-liner:

```
- [story-03/work] playwright needs `npx playwright install` before first run
- [story-04/review] worker used toBeDefined — must assert specific values
- [story-05/work] port 3000 was already in use, kill prior process first
```

### Who Writes Learnings

- **work** — appends when it hits errors, gotchas, or surprising
  behavior during implementation
- **review** — appends when it catches a pattern of mistakes (so
  the worker avoids it on the next story, not just the retry)

### The Feedback Loop

```
Story-03 attempt 1:
  [work]   makes mistake X
  [review]  catches it, writes REVISE + appends to learnings.md

Story-03 attempt 2:
  [work]   reads learnings.md, reads review-feedback.txt, fixes X

Story-04 attempt 1:
  [work]   reads learnings.md, avoids mistake X from the start
```

Without learnings, the Story-04 worker would have no memory of
what went wrong in Story-03 — each story starts a fresh agent
instance.


## Configuration

### Environment Variables

| Variable               | Default | Description                                               |
| ---------------------- | ------- | --------------------------------------------------------- |
| `GYRO_MAX_RETRIES`    | `5`     | Max work+review cycles per story                          |
| `GYRO_DEFAULT_AGENT`  | `auto`  | Fallback agent when models config has no `agent:` prefix  |
| `GYRO_BASE_BRANCH`    | `main`  | Target branch for pull requests                           |

### Agent Selection

Each step's agent is set in `prd.json` via the `"agent:model"` format
(see [Model Selection](#model-selection)). The `GYRO_DEFAULT_AGENT`
env var is only used as fallback for bare aliases (e.g., `"opus"`
instead of `"claude:opus"`).

```bash
# Auto-detect default (tries claude first, then codex)
./gyro.sh

# Force default to codex for steps without explicit agent
GYRO_DEFAULT_AGENT=codex ./gyro.sh
```

### Examples

```bash
# Fewer retries
GYRO_MAX_RETRIES=3 ./gyro.sh
```

Note: Per-step agent and model is configured in `prd.json`.
See [Model Selection](#model-selection).

### AGENTS.md

This file is read by every agent instance. Keep it focused:

```markdown
# Project: My App

## Tech Stack
- Backend: Go, SQLite
- Frontend: React, TypeScript, Vite

## Quality Gates (must ALL pass before committing)
cd backend && go test ./... -v -count=1
cd frontend && npx playwright test

## Rules
- Backend: TDD. Write failing test FIRST.
- NEVER delete or skip existing tests.
- NEVER use hardcoded return values.
```

Key principles from OpenAI's harness engineering:
- Keep AGENTS.md short — it's a TABLE OF CONTENTS, not an encyclopedia
- A giant instruction file crowds out the actual task context
- Point to deeper docs instead of putting everything inline


## Extending with Custom Steps

### Adding a New Step

1. Create a prompt file at `.gyro/prompts/{step-name}.md`
2. Add the step to a pipeline in `prd.json`

### Example: Security Scan Step

`.gyro/prompts/security-scan.md`:
```markdown
You are a security auditor. You did NOT write this code.

## Startup
1. Read .gyro/state/current-story.txt for the story
2. Read the changed files via git diff HEAD~1

## Check for:
- [ ] SQL injection (raw string concatenation in queries)
- [ ] XSS (unescaped user input in HTML)
- [ ] Missing input validation at API boundaries
- [ ] Hardcoded secrets or credentials
- [ ] Path traversal vulnerabilities
- [ ] Missing authentication/authorization checks

## Decision
If no security issues:
  echo "SHIP" > .gyro/state/review-result.txt

If issues found:
  echo "REVISE" > .gyro/state/review-result.txt
  Write findings to .gyro/state/review-feedback.txt
```

`prd.json`:
```json
"pipelines": {
  "secure-api": ["work", "review", "security-scan"]
}
```

### Example: Performance Benchmark Step

`.gyro/prompts/perf-check.md`:
```markdown
You are a performance engineer.

## Steps
1. Read the story's acceptance criteria for performance targets
2. Write a benchmark test if one doesn't exist
3. Run the benchmark: go test -bench=. -benchmem ./...
4. Verify results meet the targets

## Decision
If performance targets met:
  echo "SHIP" > .gyro/state/review-result.txt
If not:
  echo "REVISE" > .gyro/state/review-result.txt

```

`.gyro/prompts/a11y-check.md`:

You are an accessibility auditor.

2. Run: npx playwright test --project=a11y

3. Check for:
   - Missing alt text on images
   - Missing form labels

## Troubleshooting

### Story keeps failing with REVISE

**Check the feedback**: Read `.gyro/state/review-feedback.txt` for
the reviewer's complaints.

**Common causes**:
- Acceptance criteria too vague → reviewer can't verify, keeps saying REVISE
- Task too large → agent runs out of context before finishing
- Missing dependencies → agent can't install packages or run tests

**Fix**: Decompose the story into smaller pieces, or make the
acceptance criteria more specific.

### Agent writes fake tests (assert true)

The review step should catch this, but if it doesn't:
1. Tighten `review.md` — add explicit check patterns to flag
2. Add mutation testing as a quality gate in AGENTS.md:
   ```
   npx stryker run --mutate 'src/**/*.ts'  # mutation score > 80%
   ```
3. Add branch coverage thresholds:
   ```
   go test ./... -coverprofile=cover.out
   go tool cover -func=cover.out  # check percentages
   ```

### Agent doesn't follow TDD

Strengthen the TDD instruction in `work.md`:
```
STRICT TDD PROTOCOL:
1. Write the test file FIRST. Commit it: "test: description"
2. Run the test. It MUST FAIL. If it passes, your test is wrong.
3. Write the minimum implementation to make it pass.
4. Commit: "feat: description"
```

### AI CLI permission issues

**Claude Code**: Uses `--dangerously-skip-permissions` by default.

**Codex CLI**: Uses `-s danger-full-access` by default.

Both can be adjusted by editing `build_ai_cmd()` in gyro.sh.

### Visual verify fails (no browser MCP)

If you don't have a browser MCP tool configured yet, temporarily
remove visual-verify from frontend pipelines:

```json
"frontend-e2e": ["work", "review"]
```

Add it back when your MCP setup is ready.

### Checkpoint simplify changes too much

If the simplify pass is too aggressive:
- Add to `simplify.md`: "Only simplify if the improvement is obvious
  and unambiguous. When in doubt, leave the code as-is."
- Or remove `on_complete` if the final pass isn't needed:
  ```json
  "checkpoints": { "simplify": { "after": ["story-06"] } }
  ```

### Loop runs forever / burns API credits

Set a lower retry limit:
```bash
GYRO_MAX_RETRIES=2 ./gyro.sh
```

The script exits with error code 1 when retries are exhausted,
so you won't burn credits indefinitely.

## Converting a Plan to Gyro Tasks

If you already have an implementation plan (plan.md), you can
convert it to gyro prd.json automatically instead of writing
stories by hand. You can generate a plan with `./gyro.sh --plan`
(which reads specs and code, then writes `PLAN.md`), or write
one by hand.

### Option A: Slash Command (inside Claude Code)

```bash
# In Claude Code, run:
/plan-to-gyro plan.md
/plan-to-gyro IMPLEMENTATION_PLAN.md
/plan-to-gyro                          # auto-finds plan.md in cwd
```

This reads your plan, the existing prd.json, and AGENTS.md, then
rewrites prd.json with properly decomposed stories.

### Option B: Standalone Script

```bash
# Basic usage — reads plan.md, writes .gyro/prd.json
./plan-to-gyro.sh plan.md

# Preview without writing (dry run)
./plan-to-gyro.sh plan.md --preview

# Custom output path
./plan-to-gyro.sh plan.md --output my-prd.json

# Specify tech stack (helps with pipeline assignment)
./plan-to-gyro.sh plan.md --tech "Go+React"

# Auto-detect plan file (looks for plan.md, PLAN.md, IMPLEMENTATION_PLAN.md)
./plan-to-gyro.sh
```

### What It Does

1. Reads your plan file
2. Reads existing prd.json (preserves completed stories)
3. Reads AGENTS.md (for tech stack context)
4. Sends everything to the agent with conversion rules
5. Outputs a valid prd.json with:
   - Properly decomposed stories (one context window each)
   - Correct pipeline assignments (backend-tdd, frontend-e2e, etc.)
   - Specific, verifiable acceptance criteria
   - Checkpoints at natural boundaries
6. Shows a summary table

### Tips for Good Conversion

- **Be specific in your plan** — vague plans produce vague acceptance criteria
- **List endpoints explicitly** — "POST /items with fields x, y, z" converts
  better than "CRUD for items"
- **Mention validation rules** — "title required, max 200 chars" becomes
  specific test criteria
- **Describe UI states** — "empty state shows X, loading shows Y, error shows Z"
  converts into clear visual verification criteria
- **Review after conversion** — always run `./gyro.sh --dry-run` to verify
  the stories make sense before running the loop
